## 0. 一句话结论

24 篇 world-model 论文以 `max_concurrent=2` 跑完:**done=11,failed=13**(用时 321 分钟)。
**13 个失败不是单一系统性原因,而是 6 类**;其中**头号问题(6 篇,占失败的 46%)是 writer 在"乱锚定"**——
它把 MinerU MD 里的**作者署名块(`<sup>` 标记)和整张 HTML 表格**当作锚点 quote,这种含标记/转义的
quote 在 G3 的 `resolves_in_md` 归一化子串匹配里必然失败,于是 G3 硬阻断。**问题源头是 writer 的锚定纪律,
不是 Gate 太严。** 修掉头三类(① writer 锚定、② `also_depends_on` 字符串、⑥ survey Seal-1)预计可救回 ~9/13。

附带放大器:`config/audit.yaml` 的 `max_gate_rounds: 1` = **零重试**,任何单个 hard-finding 直接判死。
很多失败(尤其 ①④⑤)正是"带反馈重试一次就能过"的情形。

## 2. 六类根因(证据 + 修复方向)

### ① writer 锚定到 HTML 垃圾 —— 6 篇,头号问题

**论文**:2603.00825, 2604.01765, 2402.15391, 2503.19755, 2507.00603, 2405.12399
**现象**(`_failed/2026-06-08_*` 实摘):
- `branch1 anchor (quote: 'Anmol Agarwal<sup>1,2\\</sup> Sumer Singh<sup>2∗</sup> ...Overworld AI') does not resolve`
- `branch1 anchor (quote: 'Yang Zhou<sup>1,2</sup> Xiaofeng Wang<sup>1</sup> ...') does not resolve`
- `branch1 anchor (quote: '<table><tr><td>Game</td><td>Random</td>...(整张 Atari 表)</table>') does not resolve`
- `'<table>...PWM (Zhao et al.,2025)</td><td>NeurIPS&x27;25</td>...' does not resolve`

**问题源头**:writer(`scripts/llm/writer.py` → `seams.write_report`,经 opencode)给锚点 quote 填了
**MinerU MD 里的原始 HTML**——作者署名块(`<sup>` 上标)、整张 `<table>` 数据表,还带 `\\`、`&x27;`
这类转义/坏实体。G3 的 `scripts/audit/anchor_resolution.py:resolves_in_md` 用 `_normalize`(只折叠空白 +
转小写)做子串匹配,标记/实体/转义一漂移就匹配不上 → 判 unresolvable → 硬阻断。

**为什么 branch1 lint 没先拦住**:`scripts/output/anchor_lint.py` 的 `QUOTE_WORD_CAP=25` 按空白切词;
**HTML 表格无空格**(`Game</td><td>Random` 整段算 ~1 个"词"),25 词上限被绕过 → lint 放行 → 留给 G3 拦。
而且 writer **根本不该锚定作者列表/表格**——那不是实证业绩句。

**修复方向(分层,a 是主)**:
- **a. writer 纪律**(`scripts/llm/writer.py`):锚点 quote 只取 MD 里**短小、干净的散文/数字片段**;
  禁止把作者署名块、整表当 quote;选 quote 前 strip HTML 标记与实体。非实证内容(作者、致谢)不锚定。
- **b. lint 加固**(`anchor_lint.py`):25 词上限按 **HTML-strip 后**计词;含 `<table>`/`<sup>`/标签的 quote 直接判违规。
- **c. 解析容差**(`anchor_resolution.py:_normalize` / `resolves_in_md`):匹配前对**两侧**都 strip HTML 标签 +
  解码实体 + 归一转义(容忍轻微标记漂移)。这是次要兜底,主修在 a。

### ② analyzer 把 `also_depends_on` 写成字符串 —— 2 篇

**论文**:2408.14197, 2509.24527(被误归为 `stalled`,实为 branch2 Seal-1 崩溃)
**现象**:`branch2 failed Seal-1: trace: E4: also_depends_on references unknown node 'D'; ... 'Q'; ... '2'`
**问题源头**:analyzer LLM 偶发把 `also_depends_on` 输出成**字符串 `"D1"`** 而非列表 `["D1"]`;
`scripts/output/ara_schema.py:102` `for dep in node.get("also_depends_on", []) or []` **对字符串逐字符迭代**,
`"D1"` → `'D','1'` → 两个假"未知节点" → Seal-1 报错 → branch2 抛 ValueError → spoke 崩(被 `_run_spoke_guarded`
归类 `stalled`,分类也不对)。

**修复方向**:
- **ara_schema.py:102 之前 coerce**:`deps = node.get("also_depends_on") or []; if isinstance(deps, str): deps = [deps]`。
- analyzer prompt 加固(`scripts/llm/analyzer.py:165` 那条已写"必须是列表",但 LLM 偶发违反;可在 `jsonparse`/
  `branch2_ara` 解析处做规范化兜底)。
- 健壮性:branch2 Seal-1 失败应返回**干净的 gate 失败**(`audit_block`),不该抛 ValueError 崩成 `stalled`。

### ③ G2 数字不在 MD —— 2 篇

**论文**:2407.01392(`'113.9' 不在 MD,近似值 '119.5'`)、2606.02800(`'93.34' 不在,近似 '86'`)
**问题源头**:analyzer 转写了一个 skeptic 多数票无法在 MD 里确认的数字。**两种可能,需在另一台机器上证伪**:
(a) 真转写错误(analyzer 读错表格)→ 正确拦截;(b) 数字其实在某张 MD 表里、skeptic 没扫到 → skeptic 漏检。
**动作**:`grep '113.9' corpus/2407.01392*/*.md`、`grep '93.34' corpus/2606.02800*/*.md`。
在 → 修 skeptic 的 MD 覆盖(`scripts/audit/g2_data_fidelity.py` + skeptic seam);不在 → 修 analyzer 数字保真。

### ④ rigor 评分低于 Weak Accept —— 1 篇

**论文**:2601.18692(`ARA Seal Level 2 grade 'Weak Reject' (mean 2.83)`)
**问题源头**:opencode rigor 判官给 2.83。可能是论文本身偏短/工程向(真低分),也可能是判官偏严。
`max_gate_rounds=1` 下无重试。优先级低;带反馈重试或二次采样大概率能过。

### ⑤ writer 漏锚定 —— 1 篇

**论文**:2511.00062
**现象**:`unanchored empirical assertion (no <!--ref--> marker): '**数据管道的"幸存者偏差"是有意为之**...'`
（多条中文 bold 实证句没加锚点)
**问题源头**:同 ① 的 writer 纪律问题——writer 没在 bold 实证句上稳定输出 `<!--ref-->`。修 writer 锚定纪律。

### ⑥ Seal-1 缺 survey 的 config 文件 —— 1 篇

**论文**:2510.20668(survey:From Masks to Worlds)
**现象**:`branch2 failed Seal-1: missing or empty file src/configs/training.md; ... model.md`
**问题源头**:`ara_schema.py` 的 ARA 完整性校验**无条件**要求 `src/configs/training.md` + `model.md`;
综述/无实验论文根本没有训练/模型配置 → Seal-1 崩。
**修复方向**:对 survey / 非实证论文放宽这两个文件为可选,或让 branch2 写占位 stub。

## 3. 修复优先级

| 优先级 | 修复 | 影响 | 改动量 |
|---|---|---|---|
| **P0** | ① writer 锚定纪律 + lint HTML-strip + 解析容差 | 救 6 篇 + ⑤ 1 篇 = 7 | 中(writer prompt + 2 处校验) |
| **P0** | ② `also_depends_on` coerce(一行)+ Seal-1 不崩 | 救 2 篇 | 小 |
| P1 | ⑥ survey 放宽 config 必需 | 救 1 篇 | 小 |
| P1 | ③ 证伪 G2 数字(grep MD)再定方向 | 2 篇 | 先调查 |
| P2 | `max_gate_rounds: 1→2/3` + **反馈制导重试**(把 G3 verdict 喂回 writer re-emit) | 放大所有可救项 | 中 |
| P2 | ④ rigor:二次采样 / 判官校准 | 1 篇 | 调查 |

**反馈制导重试(A2)** 与用户既定的 Plan B(每篇 3 轮 audit→repair→regate)同向:spoke 的 `on_reemit`
当前是盲重试(`spoke.py:223 on_reemit=lambda i: _attempt()`),应把上一轮 verdict 的 hard-findings 注入
writer/analyzer 的 re-emit prompt。**关键完整性约束:Gate 必须始终是独立裁判,不能让 LLM 优化它——
修复只针对 findings,过门仍由 G3 新评。**
